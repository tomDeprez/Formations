01

Présentation ELK et ses composants — 2h

Comprendre pourquoi ELK est devenu la référence SIEM open-source, et comment chaque brique contribue à la sécurité d'un SI.

🔍 Elasticsearch — Le moteur

Elasticsearch est un moteur de recherche et d'analyse distribué basé sur Apache Lucene. Dans un contexte SIEM, c'est lui qui stocke et indexe l'intégralité des logs, permettant des recherches full-text en millisecondes même sur des téraoctets de données.

✅ Points forts

Recherche quasi temps-réel · Scalabilité horizontale · API REST intuitive · Schéma flexible (JSON)

⚠️ Points d'attention

Consommation RAM importante · Gestion du cluster à prévoir · Sécurité à configurer manuellement (pre-8.x)

🔧 Logstash — Le transformateur

Logstash est un pipeline de traitement de données en temps réel. Il suit une logique en 3 étapes :

Étape Rôle Exemples de plugins
INPUT Réception des données brutes beats, syslog, file, jdbc, http
FILTER Parsing, enrichissement, transformation grok, mutate, geoip, date, dns
OUTPUT Envoi vers la destination elasticsearch, file, kafka, s3

📊 Kibana — L'interface de visualisation

Kibana est l'interface web d'ELK. Dans un contexte SIEM il permet de :

  • Créer des dashboards de surveillance en temps réel
  • Exécuter des requêtes KQL (Kibana Query Language) sur les données
  • Configurer des alertes basées sur des seuils ou des patterns
  • Accéder à Elastic SIEM (Security > Overview)

🏛️ SIEM vs SIEM traditionnel

Critère SIEM traditionnel (Splunk, QRadar) ELK SIEM
Coût Licences très élevées Open-source (Elastic Basic gratuit)
Flexibilité Propriétaire, difficile à adapter Hautement configurable
Scalabilité Coût lié au volume Scalable nativement
Courbe apprentissage Modérée (UI dédiée) Plus technique mais documenté
Détection MITRE ✅ ✅ (Elastic Security)

📦 Installation via APT (méthode officielle)

bash — Ajout dépôt Elastic 
# Importer la clé GPG
wget -qO - https://artifacts.elastic.co/GPG-KEY-elasticsearch \
  | sudo gpg --dearmor -o /usr/share/keyrings/elasticsearch-keyring.gpg

# Ajouter le dépôt (version 8.x)
echo "deb [signed-by=/usr/share/keyrings/elasticsearch-keyring.gpg] \
  https://artifacts.elastic.co/packages/8.x/apt stable main" \
  | sudo tee /etc/apt/sources.list.d/elastic-8.x.list

# Mise à jour et installation
sudo apt update
sudo apt install -y elasticsearch logstash kibana

⚙️ Configuration minimale Elasticsearch

YAML — /etc/elasticsearch/elasticsearch.yml 
# Nom du cluster
cluster.name: siem-lab

# Nom du nœud (unique dans le cluster)
node.name: node-01

# Dossier de stockage des données
path.data: /var/lib/elasticsearch
path.logs: /var/log/elasticsearch

# Écouter sur toutes les interfaces (lab uniquement)
network.host: 0.0.0.0

# Pour un cluster à nœud unique en lab
discovery.type: single-node

# Sécurité (activée par défaut en 8.x)
xpack.security.enabled: true
xpack.security.http.ssl.enabled: false  # désactivé en lab

Configurer Logstash pour la collecte des logs — 3h

Logstash est le cœur du pipeline d'ingestion. On va apprendre à écrire des pipelines pour collecter des logs SSH, Apache, et des events Syslog, et les envoyer vers Elasticsearch.

🏗️ Structure d'un pipeline Logstash

Logstash DSL — structure de base 
# /etc/logstash/conf.d/mon-pipeline.conf

input {
  # D'où viennent les données ?
  beats {
    port => 5044
  }
}

filter {
  # Comment transformer les données ?
  grok {
    match => { "message" => "%{SYSLOGTIMESTAMP:timestamp} ..." }
  }
}

output {
  # Où envoyer les données transformées ?
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "logs-%{+YYYY.MM.dd}"
    user => "elastic"
    password => "MON_MOT_DE_PASSE"
  }
}

🔍 Le filtre Grok — parser des logs texte

Grok est le filtre principal pour parser des logs non structurés en champs JSON. Il utilise des patterns prédéfinis.

Logstash — Pipeline SSH auth.log 
input {
  file {
    path => "/var/log/auth.log"
    start_position => "beginning"
    type => "ssh-auth"
  }
}

filter {
  # Parser les tentatives SSH
  grok {
    match => {
      "message" => "%{SYSLOGTIMESTAMP:timestamp} %{HOSTNAME:host}
        sshd\[%{NUMBER:pid}\]: Failed password for %{USER:username}
        from %{IP:src_ip} port %{NUMBER:src_port}"
    }
  }

  # Convertir l'IP en localisation géographique
  geoip {
    source => "src_ip"
    target => "geoip"
  }

  # Tagger comme tentative de bruteforce
  mutate {
    add_tag => ["ssh_failed_login"]
  }
}

output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "siem-ssh-%{+YYYY.MM.dd}"
    user => "elastic"
    password => "MON_MOT_DE_PASSE"
  }
  # Debug : afficher dans le terminal
  stdout { codec => rubydebug }
}

📝 Patterns Grok utiles en cybersécurité

Pattern Correspond à Exemple
%{IP:src_ip} Adresse IP (v4 ou v6) 192.168.1.1
%{USER:username} Nom d'utilisateur admin, root
%{NUMBER:port} Numéro de port 22, 443
%{HTTPDATE:timestamp} Date format HTTP 01/Jan/2025:12:00:00
%{WORD:method} Mot unique GET, POST
%{URI:request} URL complète /admin/login.php
%{GREEDYDATA:message} Tout le reste Message libre
TP 02 📡 Pipeline de collecte multi-sources

Créer un pipeline Logstash qui ingère simultanément des logs SSH (/var/log/auth.log) et des logs Apache (/var/log/apache2/access.log).

  • Utiliser deux blocs input file avec des type distincts
  • Utiliser des conditions if [type] == "ssh" dans le filtre
  • Ajouter geoip sur l'IP source des logs Apache
  • Créer deux indices séparés dans Elasticsearch : siem-ssh-* et siem-apache-*
🔓 Voir la solution
✅ Pipeline complet
Logstash — multi-sources.conf 
input {
  file {
    path => "/var/log/auth.log"
    type => "ssh"
    start_position => "beginning"
  }
  file {
    path => "/var/log/apache2/access.log"
    type => "apache"
    start_position => "beginning"
  }
}

filter {
  if [type] == "ssh" {
    grok {
      match => { "message" => "%{SYSLOGTIMESTAMP} %{HOSTNAME} sshd.*: %{GREEDYDATA:ssh_event}" }
    }
    mutate { add_field => { "[@metadata][index]" => "siem-ssh" } }
  }

  if [type] == "apache" {
    grok {
      match => { "message" => "%{COMBINEDAPACHELOG}" }
    }
    geoip { source => "clientip" }
    mutate { add_field => { "[@metadata][index]" => "siem-apache" } }
  }
}

output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "%{[@metadata][index]}-%{+YYYY.MM.dd}"
    user => "elastic"
    password => "MON_MOT_DE_PASSE"
  }
}
04

Indexation et recherche avec Elasticsearch — 3h

Elasticsearch stocke les données en JSON dans des "indices". Comprendre comment créer des indices, définir des mappings, et écrire des requêtes DSL est fondamental pour exploiter les données de sécurité.

📂 Concepts fondamentaux

Concept ES Analogie SQL Description
Index Table Ensemble de documents JSON similaires
Document Ligne Un enregistrement JSON (un événement de log)
Champ Colonne Une propriété du document
Mapping Schéma Définition des types de champs
Shard Partition Fragment d'index pour la distribution

🗺️ Créer un mapping personnalisé

JSON — Créer un index avec mapping SIEM 
# PUT /siem-events-000001
curl -u elastic:PASS -X PUT 'http://localhost:9200/siem-events-000001' \
  -H 'Content-Type: application/json' -d '{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0
  },
  "mappings": {
    "properties": {
      "@timestamp":  { "type": "date" },
      "src_ip":       { "type": "ip" },
      "username":     { "type": "keyword" },
      "event_type":   { "type": "keyword" },
      "message":      { "type": "text" },
      "geoip.location": { "type": "geo_point" }
    }
  }
}'

🔍 Requêtes DSL — Exemples SIEM

JSON — Recherche : tentatives SSH échouées 
# GET /siem-ssh-*/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "message": "Failed password" } }
      ],
      "filter": [
        { "range": {
          "@timestamp": { "gte": "now-24h" }
        }}
      ]
    }
  },
  "aggs": {
    "top_attackers": {
      "terms": {
        "field": "src_ip",
        "size": 10
      }
    }
  }
}
💡

KQL dans Kibana : Pour la recherche rapide dans Kibana Discover, on utilise le Kibana Query Language :
message: "Failed password" AND src_ip: "10.*"
event.type: "authentication_failure" AND @timestamp >= now-1h

Configurer des alertes et notifications dans Kibana — 4h

Les alertes transforment Kibana d'un outil de visualisation en système de détection actif. On apprend à créer des règles qui se déclenchent automatiquement sur des patterns d'attaque.

🔔 Types de règles d'alertes

Type de règle Description Cas d'usage SIEM
Threshold Seuil sur un comptage +10 échecs SSH en 5 min → alerte
KQL Alert Requête KQL + seuil Detect event.action: "login_failure"
Machine Learning Anomalie détectée par ML Pic de trafic anormal
EQL (Event Query Language) Séquences d'événements login → privilege_escalation → exfiltration
Indicator Match IOC (IP, hash, domaine) Match contre une threat intel list

⚡ Créer une règle Threshold (bruteforce SSH)

🧭

Navigation Kibana : Stack Management → Rules → Create rule → Elasticsearch query (threshold)

JSON — Règle d'alerte : bruteforce SSH 
{
  "name": "Bruteforce SSH - 10 echecs en 5 min",
  "rule_type_id": ".es-query",
  "schedule": { "interval": "1m" },
  "params": {
    "index": ["siem-ssh-*"],
    "timeField": "@timestamp",
    "timeWindowSize": 5,
    "timeWindowUnit": "m",
    "threshold": [10],
    "thresholdComparator": ">",
    "body": {
      "query": {
        "match": { "message": "Failed password" }
      }
    }
  }
}

TP 05 🚨 Créer un système d'alertes multi-niveaux

Mettre en place un système d'alertes à 3 niveaux de sévérité pour les événements SSH :

  • LOW : 3+ échecs en 10 min depuis la même IP → log dans un index siem-alerts-low
  • MEDIUM : 10+ échecs en 5 min → notification email
  • HIGH : 50+ échecs en 2 min → webhook vers Slack + email
🔓 Voir la solution
✅ Architecture d'alertes
💡

Créer 3 règles séparées dans Stack Management → Rules, chacune avec :

— Le même index source : siem-ssh-*

— La même query : message: "Failed password"

— Seuils différents : >3 / 10m, >10 / 5m, >50 / 2m

— Actions différentes par niveau (Index → Email → Email + Webhook)

🔍 Les règles de détection MITRE ATT&CK

Elastic Security embarque des centaines de règles pré-construites alignées MITRE ATT&CK. On les active depuis Security → Rules → Detection Rules.

Tactique MITRE Exemple de règle Source de log
Initial Access SSH Brute Force Detection auth.log
Credential Access /etc/passwd and /etc/shadow Discovery auditd
Persistence SSH Authorized Keys File Modification auditd
Privilege Escalation Sudo Command Enumeration via sudo -l auditd
Exfiltration Potential DNS Tunneling via nslookup dns
Defense Evasion Log Files Deleted with touch auditd

📁 Cases — Gestion formelle des incidents

JSON — Créer un Case via l'API 
curl -u elastic:PASS -X POST 'localhost:5601/api/cases' \
  -H 'kbn-xsrf: true' -H 'Content-Type: application/json' \
  -d '{
  "title": "Campagne bruteforce SSH - 2025-01-15",
  "description": "Détection de 200+ tentatives SSH depuis 45.33.32.156 entre 14h et 15h.",

TP 07 🕵️ Gérer un incident de bout en bout

Simuler un incident complet : détection → investigation → case → résolution.

  • Utiliser la simulation du TP 06 (bruteforce SSH) comme incident de départ
  • Dans Security → Alerts : changer le statut de l'alerte en "In Progress"
  • Créer une Timeline pour investiguer tous les événements liés à l'IP source
  • Créer un Case : titre, description, sévérité HIGH, tags MITRE
  • Ajouter la Timeline comme preuve au Case
  • Rédiger une note de résolution et fermer le Case

Optimisation des performances d'ELK — 3h

ELK peut ingérer des centaines de milliers d'événements par seconde, mais seulement si la stack est correctement dimensionnée et optimisée. Ce chapitre couvre les paramètres clés.

🧠 Dimensionnement mémoire Elasticsearch

bash — /etc/elasticsearch/jvm.options.d/heap.options 
# Règle d'or : heap = 50% de la RAM, max 32 Go (compressed OOPs)
# Sur une machine avec 16 Go de RAM :
-Xms8g   # heap initial = 8 Go
-Xmx8g   # heap max = 8 Go (toujours égaux !)

# NEVER dépasser 32 Go (compressed OOPs se désactivent, perf chute)

📦 Gestion du cycle de vie des indices (ILM)

L'Index Lifecycle Management permet d'automatiser la rotation, la compression et la suppression des anciens indices.

JSON — Politique ILM : hot/warm/delete 
# PUT _ilm/policy/siem-lifecycle
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "50gb",
            "max_age": "1d"
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "shrink": { "number_of_shards": 1 },
          "forcemerge": { "max_num_segments": 1 }
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": { "delete": {} }
      }
    }
  }
}

⚙️ Optimisation du pipeline Logstash

YAML — /etc/logstash/logstash.yml 
# Nombre de workers parallèles (= nb de CPU)
pipeline.workers: 4

# Taille du batch (événements traités en simultané)
pipeline.batch.size: 500

# Délai max entre deux batches (ms)
pipeline.batch.delay: 50

# File d'attente persistante (évite la perte de données)
queue.type: persisted
queue.max_bytes: 4gb

TP 08 ⚡ Mettre en place une politique ILM

Créer une politique ILM pour les indices SIEM avec une rétention de 30 jours et une rotation quotidienne.

  • Créer la politique siem-30days via l'API ou Stack Management → Index Lifecycle Policies
  • Phase hot : rollover si taille > 10 Go ou ancienneté > 1 jour
  • Phase delete : suppression après 30 jours

Déploiement en production et bonnes pratiques de sécurité — 3h

Un ELK mal sécurisé est une cible pour les attaquants. Des milliers d'instances Elasticsearch ont été compromises et leurs données effacées ou exfiltrées. Ce chapitre est non négociable.

🔐 Sécuriser Elasticsearch en production

YAML — elasticsearch.yml production-ready 
# Activer TLS pour les communications HTTP
xpack.security.http.ssl.enabled: true
xpack.security.http.ssl.keystore.path: elastic-certificates.p12

# Activer TLS pour les communications inter-nœuds
xpack.security.transport.ssl.enabled: true
xpack.security.transport.ssl.keystore.path: elastic-certificates.p12

# Ne PAS écouter sur 0.0.0.0 en production
network.host: 10.0.1.5  # IP interne seulement
http.port: 9200

# Activer l'audit log Elasticsearch
xpack.security.audit.enabled: true

bash — Créer des rôles et utilisateurs SOC
