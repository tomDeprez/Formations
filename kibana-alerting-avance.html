ℹ️
Deux systèmes d'alerting dans Kibana

Kibana Rules (Stack Management → Rules) : alertes gérées par Kibana, configurables en UI, avec connecteurs. Elasticsearch Watcher (Dev Tools) : alertes JSON bas niveau, plus flexibles, stockées dans ES. On voit les deux.

2 Types de règles disponibles

📊
Threshold Rule
Déclenche quand un comptage dépasse (ou passe sous) un seuil. Simple et efficace.
💡 Quand : trop d'erreurs 5xx, trop de requêtes/sec
⚡
ES|QL Rule
Requête ES|QL transformée en alerte. Très puissant pour les logiques métier complexes.
💡 Quand : logique multi-champs, transformations, calculs
📐
Metric Threshold
Surveille une métrique numérique sur un index. Supporte avg, sum, min, max, count.
💡 Quand : latence moyenne, CPU, taille des docs
🗺️
GeoIP + Threshold
Combine géolocalisation IP et seuil. Alerte sur des connexions depuis des pays non-autorisés.
💡 Quand : connexion depuis pays interdit, Tor, impossible travel
🤖
ML Anomaly Detection
Alerte quand le ML détecte une anomalie (pic, creux, comportement inhabituel).
💡 Quand : volume inhabituel, heure insolite, user anormal
👁️
Elasticsearch Watcher
Alertes JSON bas niveau stockées dans ES. Conditions complexes, transformations, chaînes.
💡 Quand : logique avancée, séquences, enrichissements

3 Anatomie d'une règle Kibana

Toute règle Kibana est composée de 4 blocs fondamentaux :

1. DEFINE — La requête
Que chercher ? Index pattern, requête KQL ou ES|QL, champs à surveiller.
2. WHEN — La condition
Quel seuil déclenche l'alerte ? count > 100, avg > 500ms, IS ABOVE / BELOW.
3. SCHEDULE — La fréquence
À quelle fréquence évaluer la règle ? Toutes les 1 min, 5 min, 1h. Impact performance.
4. ACTIONS — Les notifications
Quoi faire quand ça déclenche ? Email, Slack, PagerDuty, webhook, créer un ticket Jira.

4 Cycle de vie d'une alerte

ÉtatSignificationAction recommandée
ActiveLa condition est dépassée en ce momentInvestiguer immédiatement
FlappingL'alerte passe active/inactive rapidement (signal instable)Augmenter le seuil ou la fenêtre temporelle
RecoveredLa condition n'est plus remplieVérifier la résolution
OKSous le seuil — tout va bien—
ErrorErreur d'exécution de la règleVérifier la requête et l'index
💡
Bonnes pratiques générales

• Commencez par des seuils larges et affinez
• Configurez toujours une action "Recovered" pour savoir quand ça revient à la normale
• Utilisez le "throttle" pour éviter les tempêtes d'alertes
• Testez la requête dans Discover avant de créer la règle

📊 Threshold Rules

Règles de seuil —
Simple mais puissant

Le type d'alerte le plus courant. Déclenche quand un compte, une somme ou une moyenne dépasse ou passe sous un seuil défini.

1 Créer une Threshold Rule

1

Aller dans Rules

☰›Management›Stack Management → Rules → Create rule
2

Choisir le type

Sélectionnez "Elasticsearch query" dans la liste des types de règle. C'est le type threshold/query polyvalent.

3

Configurer la requête

Renseignez l'index, la requête KQL ou DSL, et la fenêtre temporelle (time window).

4

Configurer la condition

Définissez : quand déclencher (count is above, avg is above…), avec quelle valeur de seuil.

5

Configurer le schedule et les actions

Fréquence d'évaluation, connecteur de notification, message personnalisé.

⚠️
Paramètre critique : "Time window"

La fenêtre temporelle détermine sur quelle période Elasticsearch compte les events. Trop courte → faux positifs (pics normaux). Trop longue → détection lente. Règle générale : time window = 5 à 10 × le schedule.

3 Exemple — Taux d'erreur HTTP trop élevé

🎯 Objectif
Déclencher quand le pourcentage de réponses 5xx dépasse 10% du trafic total — indique une panne applicative ou surcharge.
💡
Utiliser "Elasticsearch query" avec agrégation custom

Pour un ratio, on ne peut pas utiliser directement une Threshold Rule simple. On utilisera une ES|QL Rule ou une requête DSL avec script. Voir la section ES|QL pour l'implémentation recommandée.

Approche DSL — agrégation + scriptDSL
// Utilisé comme "custom query" dans une ES|QL Rule ou Watcher
GET /tp-nginx/_search
{
  "size": 0,
  "aggs": {
    "total": { "value_count": { "field": "http.response.status_code" } },
    "errors_5xx": {
      "filter": { "range": { "http.response.status_code": { "gte": 500 } } }
    },
    "taux_erreur": {
      "bucket_script": {
        "buckets_path": { "errors": "errors_5xx._count", "total": "total" },
        "script": "params.errors / params.total * 100"
      }
    }
  }
}
// Déclencher si taux_erreur.value > 10

4 Per-term grouping — Une alerte par entité

🎯 Objectif
Surveiller chaque serveur individuellement. Si web-01 a trop d'erreurs mais web-02 va bien → alerte uniquement pour web-01.
🏷️
Configuration Group by
Configuration du regroupement par terme
Group by (split)
Kibana évalue la condition séparément pour chaque valeur du champ choisi. Chaque groupe peut être en état Active ou OK indépendamment.
source.ip ← par IP attaquante host.name ← par serveur user.name ← par utilisateur destination.port ← par port cible http.response.status_code ← par code HTTP
Top N (nombre de groupes max)
Limite le nombre de termes évalués. Si vous avez 10 000 IPs, évaluer toutes crée une surcharge. Limitez aux top N les plus fréquentes.
Top 100 ← bon compromis Top 1000 ← pour des cas précis Top 10000 ← surcharge possible
⚡ ES|QL Rules

ES|QL Rules —
La détection la plus puissante

Écrivez n'importe quelle requête ES|QL et transformez-la en alerte. Idéal pour les logiques complexes que les Threshold Rules ne peuvent pas exprimer.

1 Comment fonctionne une ES|QL Rule

Kibana exécute votre requête ES|QL selon le schedule défini. Si la requête retourne au moins un document (ou si une colonne numérique dépasse un seuil), l'alerte se déclenche.

Mode "row count"
L'alerte déclenche si la requête retourne ≥ 1 ligne. Utilisez LIMIT 1 si vous voulez juste "est-ce que ça existe ?".
Mode "value threshold"
Comparez une valeur numérique retournée par ES|QL à un seuil. Ex: STATS n = COUNT(*) → alerte si n > 100.

2 Règle — Détection brute force SSH

🎯 Objectif
Déclencher quand une IP envoie plus de 10 tentatives d'auth SSH échouées en 5 minutes, et envoyer l'IP + le nombre dans la notification.
Requête ES|QL de la règleES|QL
// Copiez cette requête dans la règle ES|QL
FROM tp-auth
  | WHERE event.outcome == "failure"
      AND destination.port == 22
      AND @timestamp >= "now-5m"
  | STATS nb_echecs = COUNT(*) BY source.ip
  | WHERE nb_echecs > 10
  | SORT nb_echecs DESC
⚙️
Paramètres de la règle
Schedule
Every 1 minute
Condition
Row count is above 0 (si la requête retourne des lignes, il y a du brute force)
Grouping
La requête fait déjà le grouping par source.ip — une alerte par IP
Throttle
10 minutes — évite d'être spammé à chaque évaluation

Message de notification avec variables

Template Mustache — Message SlackMUSTACHE
🚨 *BRUTE FORCE SSH DÉTECTÉ*

IP Source : *{{context.value.source\.ip}}*
Nb d'échecs : *{{context.value.nb_echecs}}* tentatives en 5 min
Hôte cible : {{context.value.host\.name}}

Déclenché : {{date}}
Règle : {{rule.name}}

3 Règle — Détection d'exfiltration de données

🎯 Objectif
Alerter quand la somme des données transférées par une IP dépasse 50 Mo en 15 minutes — signature d'exfiltration.
ES|QL — Détection exfiltration volumétriqueES|QL
FROM tp-nginx
  | WHERE @timestamp >= "now-15m"
      AND http.request.method == "POST"
  | STATS
      total_bytes = SUM(http.response.body.bytes),
      nb_requetes = COUNT(*),
      max_bytes   = MAX(http.response.body.bytes)
      BY source.ip
  | EVAL total_mb = ROUND(total_bytes / 1048576, 2)
  | WHERE total_mb > 50
  | SORT total_mb DESC
💡
Affiner selon votre contexte

Adaptez le seuil (50 Mo) à votre baseline normale. Si vos uploads légitimes atteignent 200 Mo, montez le seuil ou filtrez par endpoint avec AND url.path != "/api/upload-legitime".

4 Règle — Nouvelle IP jamais vue

🎯 Objectif
Détecter une IP qui se connecte pour la première fois en dehors des heures ouvrées (entre 22h et 6h). Signature d'un accès illégitime.
ES|QL — Connexion hors-heures depuis IP inconnueES|QL
FROM tp-auth
  | WHERE event.outcome == "success"
      AND @timestamp >= "now-1h"
  | EVAL
      heure    = DATE_EXTRACT("HOUR_OF_DAY", @timestamp),
      est_nuit = CASE(heure >= 22 OR heure < 6, true, false)
  | WHERE est_nuit == true
  | STATS
      nb_logins   = COUNT(*),
      users_uniq  = COUNT_DISTINCT(user.name),
      premiere    = MIN(@timestamp)
      BY source.ip
  | SORT premiere ASC
🗺️ GeoIP & Localisation

Alertes GeoIP —
Localiser les attaquants

Enrichissez vos logs avec la géolocalisation IP, puis créez des alertes sur des pays interdits, l'impossible travel, ou des connexions depuis Tor.

1 Pipeline GeoIP — Enrichir à l'ingestion

☰›Management›Dev Tools → Console
Créer le pipeline d'enrichissement GeoIPDSL
PUT /_ingest/pipeline/geoip-enrichment
{
  "description": "Enrichit source.ip et destination.ip avec GeoIP",
  "processors": [

    // ─ Géolocaliser l'IP source ──────────────────────
    {
      "geoip": {
        "field": "source.ip",
        "target_field": "source.geo",
        "properties": [
          "country_iso_code", "country_name",
          "city_name", "region_name",
          "location",    // {lat, lon} pour les cartes
          "autonomous_system_number",
          "autonomous_system_organization"
        ],
        "ignore_missing": true,
        "ignore_failure": true
      }
    },

    // ─ Ajouter un tag si IP privée (RFC1918) ────────
    {
      "set": {
        "if": "ctx.source?.ip =~ /^(10\\.|172\\.(1[6-9]|2[0-9]|3[01])\\.|192\\.168\\.).*/",
        "field": "source.is_private",
        "value": true
      }
    },
    {
      "set": {
        "if": "ctx.source?.ip != null && ctx.source?.is_private == null",
        "field": "source.is_private",
        "value": false
      }
    },

    // ─ Normaliser le code pays en majuscules ────────
    {
      "uppercase": {
        "field": "source.geo.country_iso_code",
        "ignore_missing": true
      }
    }
  ]
}
Appliquer le pipeline sur les nouveaux docsDSL
// Option 1 : Dans logstash.conf
output {
  elasticsearch {
    hosts => ["localhost:9200"]
    pipeline => "geoip-enrichment"
    index => "logs-%{+YYYY.MM.dd}"
  }
}

// Option 2 : Index template pour appliquer automatiquement
PUT /_index_template/logs-template
{
  "index_patterns": ["logs-*", "nginx-*"],
  "template": {
    "settings": {
      "default_pipeline": "geoip-enrichment"
    }
  }
}

// Option 3 : Réindexer les données existantes
POST /tp-nginx/_update_by_query?pipeline=geoip-enrichment
{ "query": { "match_all": {} } }
💡
Vérifier que GeoIP fonctionne

Après le pipeline, vérifiez avec GET /tp-nginx/_search que les docs ont bien un champ source.geo.country_iso_code. S'il est absent sur les IPs privées, c'est normal (GeoIP ne gère pas le RFC1918).

2 Alerte — Connexion depuis un pays interdit

🎯 Objectif
Déclencher immédiatement si une connexion réussie est enregistrée depuis un pays non autorisé (ex: Russie, Corée du Nord, Iran).
ES|QL Rule — Pays interditsES|QL
FROM tp-auth,logs-*
  | WHERE event.outcome == "success"
      AND @timestamp >= "now-5m"
      AND source.is_private == false

  // Liste de pays interdits (ISO 3166-1 alpha-2)
  | WHERE source.geo.country_iso_code IN (
      "RU", "KP", "IR", "CN", "BY",
      "SY", "CU", "VE", "MM"
    )

  | KEEP
      @timestamp,
      source.ip,
      source.geo.country_iso_code,
      source.geo.country_name,
      source.geo.city_name,
      source.geo.location,
      user.name,
      event.action

  | SORT @timestamp DESC

3 Alerte — Impossible Travel

🎯 Objectif
Détecter qu'un utilisateur s'est connecté depuis Paris à 10h, puis depuis Tokyo à 10h15 — physiquement impossible. Indique un compte compromis.
⚠️
Impossible travel — implémentation

Ce pattern nécessite de calculer la distance géographique entre deux connexions successives du même utilisateur et de la diviser par le temps écoulé pour obtenir une "vitesse". Si > 1000 km/h → impossible. En ES|QL pur, cela nécessite un Watcher avec script Painless ou une enrichment policy.

Watcher — Impossible Travel avec script PainlessWATCHER JSON
PUT /_watcher/watch/impossible-travel
{
  "trigger": { "schedule": { "interval": "5m" } },
  "input": {
    "search": {
      "request": {
        "indices": ["tp-auth"],
        "body": {
          "size": 0,
          "query": {
            "bool": {
              "filter": [
                { "term": { "event.outcome": "success" } },
                { "range": { "@timestamp": { "gte": "now-1h" } } },
                { "exists": { "field": "source.geo.location" } }
              ]
            }
          },
          "aggs": {
            "par_user": {
              "terms": { "field": "user.name" },
              "aggs": {
                "derniere_connexion": { "top_hits": {
                  "size": 2,
                  "sort": [{ "@timestamp": "desc" }],
                  "_source": ["source.ip", "source.geo.location",
                               "source.geo.country_name", "@timestamp"]
                }}
              }
            }
          }
        }
      }
    }
  },
  "condition": {
    "script": {
      "lang": "painless",
      "source": """
        for (bucket in ctx.payload.aggregations.par_user.buckets) {
          def hits = bucket.derniere_connexion.hits.hits;
          if (hits.length < 2) continue;
          def loc1 = hits[0]._source['source.geo.location'];
          def loc2 = hits[1]._source['source.geo.location'];
          if (loc1 == null || loc2 == null) continue;
          // Distance approx en km (formule Haversine simplifiée)
          double lat1 = loc1.lat, lon1 = loc1.lon;
          double lat2 = loc2.lat, lon2 = loc2.lon;
          double dlat = Math.toRadians(lat2 - lat1);
          double dlon = Math.toRadians(lon2 - lon1);
          double a = Math.sin(dlat/2)*Math.sin(dlat/2) +
            Math.cos(Math.toRadians(lat1))*Math.cos(Math.toRadians(lat2))*
            Math.sin(dlon/2)*Math.sin(dlon/2);
          double dist_km = 2 * 6371 * Math.asin(Math.sqrt(a));
          // Temps écoulé en heures
          def t1 = hits[0]._source['@timestamp'];
          def t2 = hits[1]._source['@timestamp'];
          // Si distance > 500km → alerte (simplification)
          if (dist_km > 500) return true;
        }
        return false;
      """
    }
  },
  "actions": {
    "slack_alert": {
      "slack": {
        "message": {
          "to": ["#security-soc"],
          "text": "✈️ IMPOSSIBLE TRAVEL détecté — compte potentiellement compromis"
        }
      }
    }
  }
}

4 Alerte — Connexion depuis Tor ou VPN suspect

🎯 Objectif
Détecter des connexions depuis des nœuds de sortie Tor ou des fournisseurs de VPN/proxy connus. Indicateur fort d'anonymisation intentionnelle.
ES|QL — Détection Tor via ASNES|QL
FROM tp-nginx
  | WHERE @timestamp >= "now-15m"
      AND source.is_private == false

  // ASN des nœuds Tor et fournisseurs anonymisants connus
  | WHERE source.geo.autonomous_system_number IN (
      // Tor Project Official Exit Nodes (exemples)
      4224, 8001, 9001, 14618,
      // Mullvad VPN
      39351,
      // NordVPN
      212238,
      // ExpressVPN
      139895
    )
    OR source.geo.autonomous_system_organization LIKE "*Tor*"
    OR source.geo.autonomous_system_organization LIKE "*anonymiz*"
    OR source.geo.autonomous_system_organization LIKE "*VPN*"

  | STATS
      nb = COUNT(*),
      users = VALUES(user.name)
      BY source.ip,
         source.geo.autonomous_system_organization,
         source.geo.country_name
  | SORT nb DESC
💡
Enrichissement Threat Intel

Pour une détection Tor plus précise, utilisez un flux de threat intelligence (liste de nœuds de sortie Tor mis à jour) combiné avec le processeur enrich de Logstash ou une Enrich Policy Elasticsearch qui ajoute un flag source.is_tor: true à l'ingestion.

2 Créer un ML Job — Anomalie de trafic

☰›Machine Learning›Anomaly Detection → Create job
1

Choisir "Single Metric" ou "Multi-Metric"

Single Metric : surveille une seule valeur (ex: count global). Multi-Metric : surveille plusieurs métriques en simultané (count + bytes + latency). Population : compare les entités entre elles (ex: quelle IP est anormale par rapport aux autres).

2

Configurer le "Bucket span"

Le bucket span est la granularité temporelle d'analyse. Règle empirique : ≥ 1/12e de la période de détection souhaitée.

1m — détection en quasi temps réel 15m ← bon compromis 1h ← pour tendances horaires 1d — pour patterns journaliers
3

Définir le détecteur

Choisissez la fonction ML et les champs à analyser.

Choisissez le ML Job à surveiller (créé précédemment).
Anomaly score threshold
Le score d'anomalie ML va de 0 à 100. Plus il est élevé, plus l'anomalie est significative.
score ≥ 25 — faible anomalie (beaucoup de bruit) score ≥ 50 ← recommandé production score ≥ 75 ← haute confiance score = 100 ← extrêmement rare
Influencers
Kibana inclut dans l'alerte les champs qui ont le plus contribué à l'anomalie. Ex: si source.ip = 185.x.x.x est l'influencer principal, ça apparaît dans la notification.
👁️ Elasticsearch Watcher

Watcher — Alertes
JSON bas niveau

Pour les cas les plus complexes. Un Watcher est un document JSON stocké dans ES qui combine trigger, input, condition, transform et actions avec une flexibilité totale.

1 Structure complète d'un Watcher

Anatomie d'un WatcherWATCHER JSON
PUT /_watcher/watch/mon-watcher
{
  // ─ 1. TRIGGER ─ Quand s'exécuter ────────────────
  "trigger": {
    "schedule": {
      "interval": "5m"
      // Ou cron : "cron": "0 */5 * * * ?"
      // Ou horaire précis : "hourly": { "minute": [0, 30] }
    }
  },

  // ─ 2. INPUT ─ Récupérer les données ─────────────
  "input": {
    "search": {
      "request": {
        "indices": ["tp-nginx"],
        "body": {
          "query": {
            "range": { "@timestamp": { "gte": "now-5m" } }
          },
          "aggs": {
            "erreurs": { "filter": {
              "range": { "http.response.status_code": { "gte": 500 } }
            }}
          }
        }
      }
    }
  },

  // ─ 3. CONDITION ─ Évaluer les données ───────────
  "condition": {
    "compare": {
      "ctx.payload.aggregations.erreurs.doc_count": { "gt": 10 }
    }
    // Ou condition script Painless :
    // "script": { "source": "ctx.payload.hits.total.value > 100" }
  },

  // ─ 4. TRANSFORM ─ Modifier les données ──────────
  "transform": {
    "script": {
      "source": """
        def count = ctx.payload.aggregations.erreurs.doc_count;
        def total = ctx.payload.hits.total.value;
        def taux = total > 0 ? (count * 100 / total) : 0;
        return ['nb_erreurs': count, 'total': total, 'taux_pct': taux];
      """
    }
  },

  // ─ 5. ACTIONS ─ Notifier ─────────────────────────
  "actions": {
    "send_slack": {
      "throttle_period": "10m",
      "slack": {
        "message": {
          "to": ["#alerts-prod"],
          "text": "🔥 {{ctx.payload.nb_erreurs}} erreurs 5xx en 5min ({{ctx.payload.taux_pct}}%)"
        }
      }
    },
    "index_alert": {
      // Aussi indexer l'alerte dans ES pour l'audit
      "index": {
        "index": "alerts-log",
        "doc_id": "{{ctx.execution_time}}"
      }
    }
  }
}

2 Watcher avec conditions enchaînées

Conditions multiples avec script PainlessWATCHER JSON
PUT /_watcher/watch/multi-condition-alert
{
  "trigger": { "schedule": { "interval": "1m" } },
  "input": {
    "chain": {
      "inputs": [
        {
          // Input 1 : compter les erreurs 5xx
          "errors": { "search": { "request": {
            "indices": ["tp-nginx"],
            "body": { "query": { "bool": { "filter": [
              { "range": { "@timestamp": { "gte": "now-1m" } } },
              { "range": { "http.response.status_code": { "gte": 500 } } }
            ]}}}
          }}}
        },
        {
          // Input 2 : compter les tentatives auth
          "auth": { "search": { "request": {
            "indices": ["tp-auth"],
            "body": { "query": { "bool": { "filter": [
              { "range": { "@timestamp": { "gte": "now-1m" } } },
              { "term": { "event.outcome": "failure" } }
            ]}}}
          }}}
        }
      ]
    }
  },
  "condition": {
    "script": {
      "source": """
        int errors = ctx.payload.errors.hits.total.value;
        int authFails = ctx.payload.auth.hits.total.value;
        // Déclencher si SOIT trop d'erreurs SOIT trop d'échecs auth
        return errors > 20 || authFails > 10;
      """
    }
  },
  "actions": {
    "alert": {
      "webhook": {
        "method": "POST",
        "url": "https://hooks.slack.com/services/XXX/YYY/ZZZ",
        "body": "{\"text\": \"⚠️ Erreurs: {{ctx.payload.errors.hits.total.value}} | Auth fails: {{ctx.payload.auth.hits.total.value}}\"}"
      }
    }
  }
}

Recettes — Alertes
de production

Des règles complètes et prêtes à l'emploi pour les cas les plus courants en SOC et monitoring. Copiez, adaptez, déployez.

1 Recette — Détection SQLi multi-vecteurs

🎯 Objectif complet
Alerte dès qu'une URL contient des patterns SQLi classiques. Couvre UNION SELECT, OR 1=1, commentaires SQL, time-based blind, stacked queries.
ES|QL Rule — SQLi multi-patternES|QL
FROM tp-nginx
  | WHERE @timestamp >= "now-5m"

  // Détection des patterns SQLi dans l'URL
  | WHERE
      url.path LIKE "*UNION*SELECT*"
      OR url.path LIKE "*OR+1%3D1*"
      OR url.path LIKE "*OR 1=1*"
      OR url.path LIKE "*SLEEP(*"
      OR url.path LIKE "*BENCHMARK(*"
      OR url.path LIKE "*--*"
      OR url.path LIKE "*DROP+TABLE*"
      OR url.path LIKE "*INSERT+INTO*"
      OR user_agent.original LIKE "*sqlmap*"

  // Classifier la sévérité
  | EVAL sqli_type = CASE(
      user_agent.original LIKE "*sqlmap*", "AUTOMATED_TOOL",
      url.path LIKE "*UNION*SELECT*",             "UNION_BASED",
      url.path LIKE "*SLEEP*" OR url.path LIKE "*BENCHMARK*", "TIME_BASED_BLIND",
      "GENERIC"
    )
  | EVAL succes = http.response.status_code == 200

  | STATS
      nb_total   = COUNT(*),
      nb_succes  = COUNT_DISTINCT(url.path),
      bytes_max  = MAX(http.response.body.bytes),
      types      = VALUES(sqli_type)
      BY source.ip, source.geo.country_name

  | SORT nb_total DESC
  | LIMIT 20

2 Recette — Déviation de la baseline de trafic

🎯 Objectif
Comparer le trafic actuel à la même heure la semaine passée. Si +200% → alerte pic anormal. Si -80% → alerte perte de logs.
Watcher — Déviation baseline semaineWATCHER JSON
PUT /_watcher/watch/baseline-deviation
{
  "trigger": { "schedule": { "interval": "15m" } },
  "input": {
    "chain": { "inputs": [
      {
        // Trafic actuel (15 dernières minutes)
        "current": { "search": { "request": {
          "indices": ["tp-nginx"],
          "body": { "query": { "range": {
            "@timestamp": { "gte": "now-15m" }
          }}}
        }}}
      },
      {
        // Baseline : même créneau il y a 7 jours
        "baseline": { "search": { "request": {
          "indices": ["tp-nginx"],
          "body": { "query": { "range": {
            "@timestamp": {
              "gte": "now-7d-15m",
              "lte": "now-7d"
            }
          }}}
        }}}
      }
    ]}
  },
  "condition": {
    "script": {
      "source": """
        long current  = ctx.payload.current.hits.total.value;
        long baseline = ctx.payload.baseline.hits.total.value;
        if (baseline == 0) return false;
        double ratio = (double) current / baseline;
        // Alerte si +200% (DDoS) OU -80% (perte logs)
        return ratio > 3.0 || ratio < 0.2;
      """
    }
  },
  "transform": {
    "script": """
      long c = ctx.payload.current.hits.total.value;
      long b = ctx.payload.baseline.hits.total.value;
      double r = b > 0 ? (double)c/b*100 : 0;
      return ['current': c, 'baseline': b, 'ratio_pct': Math.round(r)];
    """
  },
  "actions": {
    "notify": {
      "slack": {
        "message": {
          "to": ["#alerts-ops"],
          "text": "📊 Déviation trafic : {{ctx.payload.current}} req vs {{ctx.payload.baseline}} (baseline) = {{ctx.payload.ratio_pct}}%"
        }
      }
    }
  }
}

3 Recette — Corrélation multi-source (Kill Chain)

🎯 Objectif
Déclencher une alerte CRITIQUE quand une IP cumule plusieurs comportements suspects : scan + auth failures + SQLi. Reconstruction de la kill chain en temps réel.
ES|QL Rule — Kill chain correlationES|QL
// Étape 1 : Agrégez les comportements par IP sur les 30 dernières minutes

FROM tp-nginx, tp-auth, tp-syslog
  | WHERE @timestamp >= "now-30m"
      AND source.ip IS NOT NULL
      AND source.is_private != true

  // Taguer chaque type d'événement suspect
  | EVAL is_scan     = url.path LIKE "*wp-login*" OR url.path LIKE "*.env*" OR url.path LIKE "*phpmyadmin*"
  | EVAL is_sqli     = url.path LIKE "*UNION*" OR user_agent.original LIKE "*sqlmap*"
  | EVAL is_bruteforce= event.outcome == "failure" AND destination.port == 22
  | EVAL is_blocked   = event.action == "denied"

  // Compter chaque type par IP
  | STATS
      nb_scan       = COUNT_DISTINCT(url.path),
      has_sqli      = COUNT(CASE(is_sqli, 1, null)),
      has_brute     = COUNT(CASE(is_bruteforce, 1, null)),
      has_blocked   = COUNT(CASE(is_blocked, 1, null)),
      country       = VALUES(source.geo.country_name),
      total_events  = COUNT(*)
      BY source.ip

  // Calculer un score de menace composite
  | EVAL threat_score =
      (CASE(nb_scan > 5, 25, 0)) +
      (CASE(has_sqli > 0,  40, 0)) +
      (CASE(has_brute > 5, 30, 0)) +
      (CASE(has_blocked > 0, 5, 0))

  // N'alerter que si score > 50 (au moins 2 types d'attaque)
  | WHERE threat_score > 50

  | SORT threat_score DESC
  | LIMIT 10
🚨
Cette règle est votre alarme principale

Un score > 50 signifie qu'une IP combine au moins deux vecteurs d'attaque différents en 30 minutes. C'est le signal d'une intrusion organisée, pas du scan aléatoire. Configurez une action PagerDuty CRITICAL et pas juste un Slack.
