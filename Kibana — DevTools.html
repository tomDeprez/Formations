GUIDE KIBANA — GROK & LOGSTASH

Kibana comme
environnement de dev

Kibana intègre deux outils essentiels : le Grok Debugger pour tester vos patterns en temps réel, et le Logstash Pipeline Manager pour déployer vos configs sans SSH.

Pourquoi utiliser Kibana pour ça ?

Avant Kibana, le cycle d'un pipeline Logstash était : modifier le .conf, redémarrer le service, lire les logs d'erreur… Kibana centralise tout dans une interface graphique.

🔬
Grok Debugger
Tester un pattern Grok sur une ligne de log brute sans pipeline actif. Affiche les champs extraits instantanément.
⚙️
Pipeline Manager
Créer, modifier et déployer des pipelines Logstash depuis Kibana, stockés dans ES — déploiement à chaud sans restart.
📊
Logstash Monitoring
Visualiser events/sec, erreurs et latence des pipelines actifs dans des dashboards intégrés.
🛠️
Dev Tools Console
Écrire des requêtes Elasticsearch directement pour vérifier que vos documents sont bien parsés.

03 Tester pas-à-pas

Exemple : log nginx avec SQLi

Sample Data — coller dans KibanaLOG
185.220.101.47 - - [15/Jan/2024:14:22:03 +0100] "GET /api/v1/users?id=1 UNION SELECT username,password HTTP/1.1" 200 12048 "-" "sqlmap/1.7.8#stable"
1

Commencer par l'IP seule

Ne cherchez pas à tout capturer d'un coup. Commencez simple.

Pattern v1GROK
%{IP:src_ip}

→ Kibana renvoie src_ip: "185.220.101.47" ✓

2

Ajouter la date HTTP

Pattern v2GROK
%{IP:src_ip} - - \[%{HTTPDATE:timestamp}\]

Les \[ et \] échappent les crochets (caractères regex spéciaux).

3

Ajouter la requête HTTP

Pattern v3GROK
%{IP:src_ip} - (?:%{USER:auth}|-) \[%{HTTPDATE:timestamp}\] "%{WORD:method} %{NOTSPACE:uri} HTTP/%{NUMBER:http_ver}"

Le groupe (?:...|-) rend le champ auth optionnel.

4

Pattern final avec status + user-agent

Pattern finalGROK
%{IP:src_ip} - (?:%{USER:auth}|-) \[%{HTTPDATE:timestamp}\] "(?:%{WORD:method} %{NOTSPACE:uri}(?:HTTP/%{NUMBER:http_ver})?|-)" %{NUMBER:status:int} (?:%{NUMBER:bytes:int}|-) %{QS:referrer} %{QS:useragent}

Le :int après status et bytes convertit automatiquement en entier dans ES.

💡
Résultat attendu

Kibana affiche src_ip, timestamp, method, uri, status (integer), bytes, referrer, useragent.

Les custom patterns Kibana sont valables uniquement pour la session browser. En production, créez /etc/logstash/patterns/custom.grok et référencez-le avec patterns_dir.

Technique d'isolation

Déboguer par bisectionGROK
# 1. Tester uniquement le timestamp :
%{SYSLOGTIMESTAMP:ts}
# → OK ? Continuer vers la droite.

# 2. Ajouter hostname :
%{SYSLOGTIMESTAMP:ts} %{HOSTNAME:host}
# → FAIL ? Séparateur tab au lieu d'espace ?
# → Essayer : %{SYSLOGTIMESTAMP:ts}\s+%{HOSTNAME:host}

# 3. Continuer jusqu'à trouver le point de blocage.

Manager

⚠️
Licence requise

Pipeline Manager → licence Gold/Enterprise ou trial 30 jours. Vérifiez : Stack Management → License Management.

01 Activer xpack dans Logstash

/etc/logstash/logstash.ymlYAML
# Activer la gestion centralisée
xpack.management.enabled: true
xpack.management.elasticsearch.hosts: ["https://localhost:9200"]
xpack.management.elasticsearch.username: "logstash_system"
xpack.management.elasticsearch.password: "votre_password"

# Quels pipelines récupérer (liste ou glob)
xpack.management.pipeline.id:
  - "main"
  - "nginx-pipeline"
  - "security-*"

# Fréquence de polling
xpack.management.logstash.poll_interval: 5s
Redémarrer et vérifierBASH
# Redémarrer
sudo systemctl restart logstash

# Suivre les logs
sudo journalctl -u logstash -f
# Chercher : [INFO] xpack.management ... Pipeline retrieved from Elasticsearch

Bon à savoir

03 Créer un pipeline

1

Cliquer sur "Create pipeline"

Un formulaire s'ouvre : Pipeline ID, Description, zone de code, paramètres de performance.

2

Renseigner le Pipeline ID

Doit correspondre exactement à xpack.management.pipeline.id dans logstash.yml.

3

Coller votre configuration Logstash

Exemple minimalLOGSTASH
input {
  beats { port => 5044 }
}
filter {
  if [fields][log_type] == "nginx" {
    grok {
      match => { "message" => "%{COMBINEDAPACHELOG}" }
      tag_on_failure => ["_grokparsefailure_nginx"]
    }
    if [request] =~ /(?i)(union.select|or.1=1)/ {
      mutate { add_tag => ["sqli"] }
    }
  }
  date {
    match => ["timestamp", "dd/MMM/yyyy:HH:mm:ss Z"]
    target => "@timestamp"
  }
}
output {
  elasticsearch {
    hosts => ["localhost:9200"]
    index => "nginx-%{+YYYY.MM.dd}"
  }
}

BEST PRACTICES

Workflow dev
itératif avec Kibana

La méthode la plus efficace : Grok Debugger en premier, Pipeline Manager ensuite, Discover pour valider.

cycle Les 5 étapes

1. Sample log

Copier une vraie ligne depuis journalctl, tail -f ou Kibana Discover.

2. Grok Debugger

Tester et affiner le pattern itérativement. Utiliser Custom Patterns si besoin.

3. Pipeline Kibana

Créer/modifier dans Stack Management → Logstash Pipelines. Déployé à chaud.

4. Discover

Vérifier les events récents et s'assurer que les champs sont bien créés et typés.

5. Monitoring

Stack Monitoring → Logstash pour valider le débit et détecter les _grokparsefailure.

discover Chercher les parse failures

☰ Menu›Analytics›Discover
1

Sélectionner le bon index pattern

En haut à gauche de Discover, choisissez l'index pattern de votre pipeline (ex: nginx-* ou logs-*).

2

Filtrer avec KQL

KQL — barre de recherche DiscoverKQL
# Tous les events avec parse failure
tags: "_grokparsefailure"

# Failure spécifique nginx
tags: "_grokparsefailure_nginx"

# Events sans src_ip (non parsé)
NOT _exists_: src_ip

# Failure + plage horaire
tags: "_grokparsefailure" AND @timestamp >= "2024-01-15T14:00:00"
3

Inspecter le champ "message" original

Cliquer sur un event en failure → développer le champ message → copier la ligne brute → coller dans le Grok Debugger.

💡
Conserver le champ message

Ne supprimez message avec mutate { remove_field } qu'après avoir validé que votre pipeline parse 100% des events.
